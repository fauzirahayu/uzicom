Metode Waterfall - Sistem Manajemen Jamaah Haji

Tujuan: Dokumentasi ini bertujuan untuk menjelaskan penerapan metode Waterfall dalam pengembangan Sistem Manajemen Jamaah Haji, yang mencakup pengumpulan kebutuhan stakeholder dan analisis kelayakan proyek secara sistematis dan berurutan.

Manfaat Metode Waterfall: Metode ini memastikan setiap tahap selesai sebelum melanjutkan ke tahap berikutnya, sehingga mengurangi risiko perubahan kebutuhan yang tidak terduga dan memudahkan pengendalian proyek.

1. Requirements (Pengumpulan Kebutuhan)

Tahap ini fokus pada pengumpulan dan pendokumentasian kebutuhan sistem dari stakeholder.

Software Requirements Specification (SRS) - Sistem Informasi dan Daftar Jamaah Haji

1. Pendahuluan

1.1 Tujuan

Dokumen ini menjelaskan kebutuhan perangkat lunak untuk Sistem Informasi Data Jamaah Haji. Tujuannya adalah:

  • Menyediakan sarana penyimpanan & pengelolaan data jamaah secara terpusat.
  • Mempermudah petugas dalam melakukan input, edit, hapus, dan pencarian data jamaah.
  • Memberikan akses terbatas bagi jamaah untuk melihat data mereka tanpa bisa mengubah.
  • User/Jamaah bisa mendaftarkan diri.
  • Menyediakan keamanan data dengan pembatasan akses berdasarkan role (Admin/User).
1.2 Lingkup Sistem

Sistem ini digunakan oleh:

  • Admin/Petugas Haji → dapat melakukan CRUD data jamaah, tambah admin, upload foto, jika user/jamaah tidak menjawab via WhatsApp selama 24 jam maka data akan dihapus dari web. Pesan WhatsApp: "Halo [nama], selamat Anda telah terdaftar di web kami yang bekerja sama dengan Kemenag. Untuk melihat data Anda, silakan kunjungi web kami di: https://jamaah-haji.local".
  • User/Jamaah → hanya dapat melihat data dan mencari informasi keberangkatan, lihat detail profil jamaah.

Sistem mencakup:

  • Dashboard jumlah jamaah.
  • Pencarian data jamaah.
  • Detail profil jamaah.
  • Validasi & keamanan data (login role, hashing password).
1.3 Definisi & Singkatan
  • CRUD: Create, Read, Update, Delete.
  • NIK: Nomor Induk Kependudukan.
  • SRS: Software Requirements Specification.

2. Deskripsi Umum

2.1 Perspektif Produk

Sistem ini berbasis web, dapat diakses menggunakan browser. Data tersimpan pada database terpusat dan dikelola oleh admin.

2.2 Fungsi Utama
  • Tambah jamaah baru.
  • Edit & hapus data jamaah.
  • Pencarian data jamaah berdasarkan nama, nik, no porsi.
  • Dashboard jumlah jamaah.
  • Pembuatan laporan jamaah dalam bentuk PDF/Excel.
  • Upload foto jamaah.
  • Role-based access (Admin/User).
2.3 Karakteristik Pengguna
  • Admin/Petugas: memiliki kemampuan menggunakan komputer dasar.
  • Jamaah: pengguna umum, hanya perlu bisa mencari, melihat data.
2.4 Batasan Sistem
  • Web berbasis PHP/MySQL.
  • Browser yang didukung: Chrome, Firefox, Edge.
  • Foto jamaah maksimal 2 MB (format JPG/PNG).

3. Kebutuhan Fungsional

3.1 Fungsi Admin
  • Tambah data jamaah baru.
  • Edit data jamaah.
  • Hapus data jamaah.
  • Tambah akun admin baru.
  • Kelola akun admin (lihat, edit, hapus akun admin).
  • Kelola akun pengguna (lihat, edit, hapus akun pengguna).
  • Melihat jumlah total jamaah di dashboard.
  • Pencarian data jamaah berdasarkan nama, nik, no porsi.
  • Pencarian akun admin/pengguna berdasarkan username atau email.
  • Upload foto jamaah.
  • Mencetak laporan data jamaah / generate laporan berdasarkan filter (jadwal keberangkatan, usia, jenis kelamin, dsb).
3.2 Fungsi User (Jamaah)
  • Melihat dashboard (jumlah jamaah).
  • Melihat daftar data jamaah.
  • Pencarian data jamaah berdasarkan nama, nik, no porsi.
  • Melihat detail profil jamaah.
  • Tidak dapat menambah, mengedit, atau menghapus data.

4. Kebutuhan Non-Fungsional

4.1 Validasi Data
  • NIK harus unik.
  • Format tanggal harus valid.
  • Nomor telepon hanya angka (10–15 digit).
  • Jenis kelamin: hanya "Laki-laki" / "Perempuan".
  • Foto jamaah JPG/PNG, maks 2 MB.
4.2 Keamanan
  • Login menggunakan role (Admin/User).
  • Password disimpan dengan hashing (bcrypt).
  • Backup data berkala.
4.3 Kinerja
  • Respon sistem maksimal 3 detik untuk pencarian data.
  • Dapat menampung hingga 10.000 data jamaah.
4.4 Reliability
  • Sistem harus tersedia minimal 99% (uptime).
  • Data tetap aman walau terjadi crash, berkat backup rutin.

5. Kebutuhan Data

Data Jamaah yang disimpan:

  • Nama Lengkap
  • NIK
  • No Porsi
  • Jenis Kelamin
  • Tanggal Lahir
  • Alamat
  • Nomor Telepon
  • Nomor Paspor
  • Golongan Darah
  • Penyakit Bawaan
  • Jadwal Keberangkatan
  • Tanggal Pulang
  • Foto Jamaah
  • Status
  • ID Pembimbing

Data laporan dapat berisi rekap jumlah jamaah, daftar jamaah sesuai jadwal, detail kesehatan jamaah, dsb.

6. Batasan Desain

  • Tampilan sederhana (user-friendly).
  • Navigasi jelas antara halaman admin & user.
  • Menggunakan Bootstrap atau CSS untuk UI.

2. Analysis (Analisis)

Tahap ini fokus pada analisis kelayakan dan perencanaan teknis sistem.

2.1 Analisis Kelayakan: Mengevaluasi kelayakan proyek dari berbagai aspek:
  • Teknis: Sistem memungkinkan dikembangkan menggunakan teknologi web (PHP/MySQL, HTML, CSS, Bootstrap, JS). Infrastruktur server yang dibutuhkan relatif standar (Laragon untuk pengembangan, hosting biasa untuk deployment).
  • Finansial: Biaya pengembangan rendah karena menggunakan software open source (PHP, MySQL, Bootstrap). Hanya memerlukan biaya server & domain.
  • Operasional: Admin/petugas sudah terbiasa menggunakan komputer dasar, sehingga sistem dapat dijalankan dengan mudah.
  • Jadwal: Timeline development sesuai dengan kompleksitas fitur.
2.2 Modeling & Diagram: Membuat berbagai diagram untuk memodelkan sistem:
  • Use Case Diagram - Menggambarkan interaksi aktor dengan sistem
  • Activity Diagram - Menggambarkan alur kerja sistem
  • Entity-Relationship Diagram (ERD) - Menggambarkan struktur database
  • Component Diagram - Menggambarkan arsitektur komponen sistem
  • Sequence Diagrams (Login, Register, Add Jamaah, Add Pembimbing) - Menggambarkan interaksi antar komponen
2.3 Perencanaan Teknologi: Menentukan teknologi yang akan digunakan:
  • Backend: PHP 8.1+ dengan MySQL 8.0
  • Frontend: HTML5, CSS3, JavaScript
  • Framework: Native PHP (tanpa framework khusus)
  • Database: MySQL dengan struktur tabel yang telah dibuat
  • Tools: Composer untuk dependency management, Git untuk version control
  • Server: Apache/Nginx dengan PHP
2.4 Risiko & Solusi: Identifikasi tantangan dan solusi:
  • Risiko: Mati lampu.
  • Solusi: menggunakan genset, nunggu lampu nyala kembali.
  • Risiko: Jaringan mati.
  • Solusi: Pake jaringan darurat seperti lewat hp, nunggu jaringan kembali seperti semula.
  • Risiko: Human error (admin salah input).
  • Solusi: Validasi form ketat & notifikasi konfirmasi sebelum simpan/hapus.

File-file Pendukung:

  • haji.sql - Struktur database MySQL
  • backend/database.php - Koneksi database
  • backend/query/ - Script PHP untuk operasi CRUD
  • contern/ - Halaman admin
  • pengguna/ - Halaman pengguna
  • uploads/ - Direktori untuk file upload
